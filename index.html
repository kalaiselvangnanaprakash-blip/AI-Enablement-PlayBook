Ground rules — read once

Never share secrets or live personal data

No passwords, keys, connection strings, or real customer/personal data. Share structure and anonymised samples only.

Verify before you trust or run

AI drafts; you decide. Read generated code/SQL before running, and never run unreviewed changes against production. AI can be confidently wrong.

How to Learn

A plan and approach to learn AI — properly

Getting good at AI isn't a course you finish; it's a habit you build. Learn on your real work, in small daily reps, and share what works.

The approach in one line

Learn by doing on your real tasks, a little every day, and pass on what works.

The learning loop

  • Learn — one thing at a time

    One technique or concept. Don't try to learn everything before using anything.

  • Apply — on real work, same day

    Use it immediately on an actual task, not a toy example.

  • Reflect — what worked, what didn't

    Keep the prompts that worked in a notes file.

  • Share — teach a teammate

    Explaining it cements your understanding and lifts the team.

Week 1 of 8 · Foundations

AI Foundations & Prompting Basics

Outcome — write effective prompts for any task.

What AI is & its limitsAI tools overview5-element promptWeak vs strong

Week 1 answers the question every new joiner has: how do I actually use AI at work? Language models predict the most likely next words from your input — they don't "know" facts the way people do. Always verify anything important; they can hallucinate (state wrong things confidently), and you must never paste confidential or personal data.

AI excels at

Drafting text · summarising · restructuring data into tables · explaining concepts · generating templates.

AI struggles with

Real-time data · precise calculations · confidential data (never paste it) · always being factually correct.

The 5-element prompt framework

ElementWhat it is · example
1 · RoleWho the AI should be. "You are a professional business writing coach…"
2 · TaskExactly what to do. "…review the following email and rewrite it…"
3 · ContextBackground, data, audience. "…it's going to my manager about a project delay…"
4 · FormatThe output shape. "…Subject line, 3-paragraph email, a call to action."
5 · ConstraintsLimits & tone. "…under 150 words; professional but warm."
❌ Weak prompt
Write me an email
✅ Strong prompt

You are a professional business writer. Write a concise follow-up to my manager, Priya, about a delayed task (monthly report, due yesterday, ready by EOD today). Tone: professional, brief apology. Format: subject line + 3 short paragraphs. Under 100 words.

✎ Practice this week

Run the same prompt in Claude and ChatGPT; compare tone and length; ask each "What are 5 things you cannot do?" · Then write 5 prompts with the framework: (1) summarise a long email thread into 3 bullets, (2) a weekly task-list template, (3) explain a new concept as if to a beginner, (4) a polite "I need more time" message, (5) 5 subject-line options for a project update.

Key takeaway

AI is a tool, not a replacement for your thinking. The person who directs AI well outperforms anyone who ignores it or blindly trusts it. Your job: become the expert at giving AI excellent instructions.

Week 2 of 8 · Foundations

Prompting Techniques & Iteration

Outcome — choose the right technique and refine to a perfect output.

Zero / Few / Chain-of-thoughtSummary · template · improve promptsIteration mindsetCommon mistakes

Three core techniques cover almost everything — and the real skill is iteration: your first output is a draft, not the final answer. The best users refine 2–4 times.

TechniqueHow it works · best for · example
Zero-shotAsk directly, no examples. Simple, well-defined tasks. "Summarise this paragraph in 2 sentences."
Few-shotGive 2–3 examples first. Matching a format/tone/style. "Here are 2 emails I like — write a 3rd in the same style."
Chain-of-thoughtAsk it to reason step by step. Complex decisions, multi-step analysis. "Think step by step: which option should I prioritise and why?"

Three prompt templates you'll use daily

Summary
You are a professional summariser. Summarise the following
[email / report / notes] into [3 bullets / 1 paragraph / 5 actions].
Audience: [manager / team]. Tone: concise, professional.
Text: [paste]
Template generator
You are a workplace productivity expert. Create a [template type]
for [purpose]. Include sections for: [list]. Format: clean, ready to
use, with headers and bullets. Audience: [who will use it].
Document improvement
You are a professional editor. Improve the following [email / report]
for clarity, professionalism, conciseness, and correct tone. After
improving it, list the top 3 changes you made and why.
Original: [paste your draft]

Common prompt mistakes & the fix

MistakeWhy · the fix
Too vagueNo context/format — add role, context, format, constraints.
Wrong toneTone not set — add "Tone: professional / casual / empathetic."
Output too longNo length limit — add "Under 100 words" or "3 bullets max."
Wrong formatFormat not set — add "Format as a table / numbered list / email."
Made-up factsNo source data — paste your actual data; never ask it to recall specifics.
Ignored a ruleToo many competing instructions — simplify; one clear task per section.
Not actionableTask too broad — break into smaller sub-tasks, one prompt each.

The iteration chain

Run → read → identify what's off (tone? length? format?) → in the same chat say "That was good, but please [specific fix]" → repeat until perfect → save the prompt that worked. That's how your library is born.

Sorting & filtering

  • Sort: single column (Data → A→Z); multi-level (Data → Sort → Add Level: Department, then Name); by colour. Always select the whole table first — never one column alone, or you misalign data.
  • Filter: Ctrl + Shift + L toggles dropdowns. Filter by multiple values (Ctrl-click), or Custom (Number Filters → Greater Than → 50000). Data → Clear before a new filter.
  • Pro tip: press Ctrl + T first — Tables auto-filter, auto-expand, and make formulas read =SUM(EmployeeData[Salary]) instead of =SUM(F2:F21).

ProblemFix
Extra spaces=TRIM(A2) — removes leading/trailing/double spaces.
Inconsistent case=PROPER(A2) for names; UPPER/LOWER for codes.

VLOOKUP — pull data across sheets

=VLOOKUP(lookup_value, table_array, col_index_num, FALSE)
-- FALSE = exact match (use this almost always)
=VLOOKUP(A2, MasterSheet!A:D, 3, FALSE)
-- find A2 in cols A–D, return column 3, exact match
ErrorCause · fix
#N/AValue not found — check spaces; wrap in IFERROR.
#REF!col_index bigger than the table — reduce it.
Wrong resultUsed TRUE — switch to FALSE for exact match.
Not updatingLookup column isn't first — move it to position 1, or use INDEX-MATCH.

VLOOKUP → INDEX-MATCH (the upgrade)

FeatureVLOOKUP → INDEX-MATCH
DirectionLeft-to-right only → any direction.
Column insertsBreaks → never breaks.
Speed on big dataSlower → significantly faster.
ReadabilityCryptic col number → explicit columns.
Multiple criteriaDifficult → straightforward.
=INDEX(return_column, MATCH(lookup_value, lookup_column, 0))
=INDEX(MasterSheet!D:D, MATCH(A2, MasterSheet!A:A, 0))
-- find the row where A2 is in col A, return col D of that row

COUNTIFS & SUMIFS — multi-condition

=COUNTIFS(C:C,"IT", H:H,"Active")            active IT staff
=SUMIFS(F:F, C:C,"Finance", H:H,"Active")    active Finance pay
-- SUMIFS: the sum_range comes FIRST

Pivot tables — summarise thousands of rows in seconds

  • Click your data → Insert → PivotTable → New worksheet.
  • ROWS = Department, COLUMNS = Status, VALUES = EmpID (Count) → instant cross-tab.
  • Right-click → Add Slicer → Role to filter interactively. Right-click → Refresh after data changes.

Week 6 of 8 · Communication

Professional Communication & Email

Outcome — write emails colleagues respect.

Golden rulesSubject-line formulaReply / Reply-AllAttachmentsCalendar

How you write an email tells people a great deal about your professionalism. These habits set you apart from day one.

Golden ruleWhat it means
Subject-line clarityThe subject alone says what it's about and any action needed.
One topic per emailSeparate subjects → separate emails; easier to reply and file.
State the ask up frontBusy people read the first 2 lines — put the request there.
Reply within 24 hoursEven just: "Received, will get back to you by [date]."
ProofreadRead aloud once; or ask AI: "Check this email for errors."
Disciplined CCDoes this person need to act, or just be aware?
BCC carefullyMass announcements only — never for secrecy in a conversation.
Subject-line formula
[Action / Status] + [Topic] + [Timeframe if needed]
✅ Action Required: Monthly Report Review — Due Fri 28 Feb
✅ Decision Needed: Vendor Selection for Q2
✅ Resolved: IT Ticket #4821 — Access Issue Fixed
Tags: [Action Required] [FYI] [Decision Needed] [Urgent] [Reminder]

Reply vs Reply All vs Forward

ScenarioUse
Relevant to everyone on the threadReply All
Only the sender needs itReply
Replying to a mass announcementReply (never Reply All)
Adding a new person / sharing contextForward
Sharing an email chain with someone newForward (don't CC without context)
Correcting a mistake sent to the whole teamReply All (everyone who saw it should see the fix)

Attachments

  • Mention it in the body: "Please find attached the Q3 report."
  • Name files professionally: Q3_Report_Feb2026_v2.pdf — not "report final FINAL.pdf".
  • Large files (>10MB): share a OneDrive/SharePoint link instead.
  • Check it's actually attached before sending — the #1 email mistake.
  • Never send sensitive data (passwords, personal info) as plain text.

Calendar invites & scheduling

  • Title specific ("Weekly Team Sync — Operations", not "Meeting"); set Required vs Optional deliberately; state the timezone; add the Teams link / room.
  • Body = agenda (3–5 bullets) + pre-read + expected duration; reminder 15 min before (1 day for important ones).
  • Recurring: always set an end date (never "no end date"); edit a single occurrence for exceptions; cancel well in advance with a reason.
Meeting invite body
Hi [Name], inviting you to [Meeting] on [Date] at [Time].
Agenda:
  1. [Topic 1] — [5 mins]
  2. [Topic 2] — [10 mins]
  3. AOB — [5 mins]
Pre-read: [link]. Please confirm your attendance. Thanks, [You]

Foundation 2 of 2

Context Engineering

Prompting is how you ask. Context engineering is making sure the model has the right textbook, calculator, and notes before it starts thinking. The AI knows general SQL; it doesn't know your schema, data, or rules — unless you give them.

Most bad answers are context failures, not model failures

When AI gives a wrong answer, the model is usually fine — the problem is the information it was given. A "smarter" model rarely fixes what better context would.

What fills the context window

The context window is finite — everything you send and the reply share it. Spend it on what makes the answer correct, and leave room for the output.

Why placement matters — "lost in the middle"

Models recall information best at the start and end of a long input, and can miss what's buried in the middle. Put the rule, the target, and the error where they'll be seen.

Key facts go first and last; the long reference material in between.

The principle: simple, concise, precise

S

Simple

One problem at a time.

C

Concise

The relevant table and rows — not the whole database.

P

Precise

Real names, real types, the exact spec.

Too much vs just enough

Overloaded — buries the signal
[pastes a 400-line stored procedure] "something's wrong with the totals, can you find it?"
Just enough — pinpoints it
  • The 15 lines that compute the total
  • 2 sample rows where it's wrong
  • "Expected 1,200; got 1,000."

Anonymise sample rows

Change names, IDs, and amounts. The AI only needs the shape — date formats, blanks vs NULLs — never the real data.

Build · Foundations

Master prompts in Markdown — write once, build anything

Power users don't type prompts into a chat box and lose them. They write prompts as Markdown (.md) files: structured, reusable, version-controlled, and token-efficient. It is the single most leveraged habit for building apps and agents.

Why Markdown, in one minute

⚡

Fewer, cleaner tokens

Markdown carries structure with almost no syntax overhead. The model reads headings, lists, and code fences natively — so your intent lands with less text than verbose prose.

🔁

Reusable & versioned

A .md file lives in Git: diff it, review it, roll it back, share it. Your prompt becomes a durable asset, not a throwaway message.

🎯

More reliable output

Clear sections — Goal, Features, Rules — remove ambiguity, so the model or the coding agent follows the spec instead of guessing.

$

Lower cost, bigger headroom

Every token is read — and on an API, paid for — and fills a finite context window. Lean Markdown means cheaper calls and more room for real work.

Tokens — the unit you're optimising

AI reads text as tokens, not words — roughly 4 characters ≈ 1 token (about ¾ of a word). Both your prompt and the reply count toward the context window, so leaner input leaves more room for output and lowers cost. Rich formats like .docx or .pdf carry hidden markup that wastes tokens; plain, structured Markdown spends the budget on content.

See your tokens live — the tokenizer

Paste any text and watch it split into tokens with a running count at platform.openai.com/tokenizer. Use it to trim a prompt, compare prose vs Markdown, and estimate cost and context use before you send.

Same intent, a fraction of the tokens

Verbose prose — token-heavy
Hi! I was hoping you might be able to help me out by writing a little script that would go ahead and read in a CSV file for me, and then, if it's not too much trouble, output all of that data as JSON, and please do make sure it handles any errors nicely and doesn't crash...
Structured .md — lean
  • Task: CSV → JSON converter
  • Input: path to a .csv
  • Output: JSON to stdout
  • Rules: handle missing/blank fields; never crash

Same instruction, far fewer tokens — and clearer for the model to follow.

Token-saving habits

  • Structure over prose — headings, lists, and tables instead of long paragraphs.
  • Cut the filler — drop greetings, apologies, and "if it's not too much trouble".
  • State each rule once — repetition costs tokens and confuses priority.
  • Trim examples — two sharp examples beat ten mediocre ones.
  • Reference, don't repeat — point to a section rather than pasting it twice.
  • Drop irrelevant context — give the minimum that makes the answer correct.

From User to Builder

How to build an agent

A chatbot replies. An agent acts — it decides which tools to use, does the steps in order, and knows when to stop. That's how real work gets done. Here's the whole pathway, kept simple.

The agent loop

Every agent runs the same cycle until the goal is met or a stop condition fires.

Build it in five steps

  • Define the job

    One sentence describing the outcome. Start with a single, clear, repetitive task — not "do everything."

  • Pick your path

    No-code to start: n8n, Lindy, Flowise, Botpress, CrewAI. Code when it gets specific: the Claude Agent SDK + Skills, OpenAI's Agents SDK, or frameworks like LangGraph.

  • Give it context & tools

    The data, the apps, and the rules. Start with 2–4 tools. Use MCP to connect systems through one standard instead of one-off integrations.

  • Add guardrails

    Require human approval for risky actions (writes, payments, deletes). Define "done", and set a step limit so it can't loop forever.

  • Test & refine

    Run realistic examples; watch for scope creep (asking unrelated questions, calling extra tools, continuing past the goal). Tighten the job statement and repeat.

Anatomy of an agent

🧠

Model

The reasoning engine. Right-size it — smaller models are often enough for tool-driven tasks.

📋

Instructions

Its job, its rules, and its definition of "done".

🔧

Tools

How it acts — search, run code, query a DB, call an API. Start with a few.

💾

Memory

Optional. Most first agents don't need it — add only when the task truly requires recall.

🛡

Guardrails

Approvals, limits, sandboxes. The more autonomy, the tighter these get.

🔁

The loop

Observe → reason → act → check, until done or stopped.

Crawl, walk, run

Your first job isn't a masterpiece — it's to make one useful thing work. Ship a no-code agent for one task, prove it reliably, then graduate to code and more tools. A single working agent beats five broken prototypes.

Keep it simple early

Skip persistent memory and retrieval until you genuinely need them — many beginners add complexity that their task doesn't require.

A worked example — SKILL.md

Two parts: YAML frontmatter (the metadata Claude reads to decide when to use it) and a Markdown body (the instructions it follows once triggered).

name: sql-review
description: Reviews SQL for correctness, performance, and safety.
  Use when the user shares a query or asks for a SQL review.

# SQL Review

When asked to review SQL, check in this order:
1. Set-based — flag row-by-row cursors and suggest a rewrite.
2. NULL-safe — NULLIF on division, COALESCE on outer joins.
3. Schema-valid — every table/column exists in the given schema.
4. Safety — flag any UPDATE / DELETE / MERGE without a WHERE.

## Output
A short list: issue · why it matters · the fix.

## Boundaries
Do NOT rewrite the whole query unless asked — only report issues.
Never invent column names; if the schema is missing, ask for it.

Claude decides whether to use a Skill from its description alone. Say what it does and when to use it ("Use when…"). Vague descriptions never fire; specific ones do.

Apply · Daily Work

SQL & Stored Procedures

Where most data teams spend their day — and where AI saves the most time, writing the first draft and the boilerplate so you focus on correctness and performance.

How text-to-SQL actually works

AI doesn't query your database — it writes SQL from your words. The quality hinges on one step: giving it the schema.

Schema awareness is the #1 factor

Give the AI your column lists and it's reliable on standard SELECT / JOIN / GROUP BY. Without them, it invents table and column names that don't exist.

Ask
For sales(region, sale_date, product_id, amount): total revenue per region for Q2 2026, plus each region's share of total, highest first. Set-based, SQL Server 2019.
You get

Aggregation + a window function (SUM() OVER()) — ready to validate. Then refine: "make the quarter a parameter."

Window functions — the concept made visual

The three ranking functions look alike but treat ties differently. Same scores, three different results:

ScoreROW_NUMBER()RANK()DENSE_RANK()
500111
500 (tie)211
48033 (skips 2)2 (no gap)
460443

ROW_NUMBER = always unique · RANK = ties share, then a gap · DENSE_RANK = ties share, no gap. Like Olympic medals: two golds, no silver.

Four worked patterns you'll reuse

-- 1 · Running total (cumulative revenue by month)
SUM(amount) OVER (ORDER BY month
                  ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)

-- 2 · Month-over-month change with LAG()
amount - LAG(amount) OVER (ORDER BY month)   -- prior row

-- 3 · Top N per group (top product in each region)
SELECT * FROM (
  SELECT region, product, sales,
         RANK() OVER (PARTITION BY region ORDER BY sales DESC) rnk
  FROM totals
) t WHERE rnk = 1;

-- 4 · Keep detail AND the group total side by side
amount, SUM(amount) OVER (PARTITION BY region) AS region_total

More asks that work

Customers who purchased in the last 30 days but NOT the prior 6 months.
Total sales by product category for Q1, sorted by revenue desc.
The 2nd-highest salary per department (window function, not TOP).
A pivot: months across columns, product categories down rows.

Apply · For QA

Reconciliation Testing

Compare source to target, prove the numbers tie out, prove nothing was lost or duplicated. After every ETL load you ask one question — can I trust this data? — and answer it with a handful of checks.

The reconciliation flow

The five checks, as SQL

-- 1 · Row count (completeness)
SELECT (SELECT COUNT(*) FROM source) AS src,
       (SELECT COUNT(*) FROM target) AS tgt;

-- 2 · Amount totals (consistency)
SELECT SUM(amount) FROM source;      -- must equal target's

-- 3 · Missing / extra keys (EXCEPT both ways)
SELECT id FROM source EXCEPT SELECT id FROM target;  -- missing
SELECT id FROM target EXCEPT SELECT id FROM source;  -- extra

-- 4 · Duplicate keys within the load (uniqueness)
SELECT id, COUNT(*) FROM target GROUP BY id HAVING COUNT(*) > 1;

-- 5 · Field-level mismatch (FULL OUTER JOIN)
SELECT s.id, s.amount AS src_amt, t.amount AS tgt_amt
FROM source s FULL OUTER JOIN target t ON s.id = t.id
WHERE s.amount <> t.amount OR s.id IS NULL OR t.id IS NULL;

What a result sheet looks like

CheckSourceTargetResult
Row count10,00010,000PASS
Sum of amount4,812,9004,812,900PASS

Duplicate keys0 duplicatesPASS

A threshold (e.g. target > 5% below source) can auto-fail the pipeline before bad data spreads.

ConsistencyTotal amount in source = total in target, per key.
UniquenessKeys appearing more than once within a load.
IntegrityTarget rows whose foreign key has no parent (orphans).
TimelinessLoads that landed late, or periods missing expected rows.

SQL Errors & Fixes

An error message is the most useful thing you can give an AI. Paste the exact text plus the code, and you usually get the cause and fix faster than searching.

The debugging loop

Copy this shape
Error: [paste the FULL error, including the Msg number]
Code that produced it: [just the relevant lines]
Environment: SQL Server 2019. Expected: [intended result].
Most likely cause first, then the exact fix. Rank if several.

MessageCause · fix

String or binary data would be truncatedValue too long for the column. Widen the column, or trim/validate input. SQL Server 2019 names the offending column for you.
Msg 8134 — divide by zeroA denominator is 0. Use NULLIF(denominator, 0) so the row returns NULL instead of erroring.
Msg 1205 — deadlock victimTwo transactions locked resources in opposite order. Access tables in a consistent order, keep transactions short, add indexes, and add retry logic.

A real before / after

You paste

Divide by zero error encountered.

SELECT region, sales/target AS pct FROM kpi;
You get back

"A target of 0 causes it. Guard the denominator:"

SELECT region,
  sales / NULLIF(target,0) AS pct
FROM kpi;

The Priority Matrix — decide what to do now

Every task is some mix of urgent and important. The quadrant tells you the action.

Urgent + Important

Do first

Crises, today's deadlines, a manager's ask. Example: report due in 2 hours.

Not urgent + Important

Schedule

Planning, learning, deep work. Example: build your tracker template.

Urgent + Not important

Delegate / minimise

Interruptions, some meetings. Example: a "quick chat" that someone else could field.

Not urgent + Not important

Eliminate

Busywork and scrolling. Example: reorganising your desk again.

Your manager can't see inside your head. This four-block structure builds trust faster than almost anything.
