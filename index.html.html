Ground rules — read once

Never share secrets or live personal data

No passwords, keys, connection strings, or real customer/personal data. Share structure and anonymised samples only.

Verify before you trust or run

AI drafts; you decide. Read generated code/SQL before running, and never run unreviewed changes against production. AI can be confidently wrong.

You own the output

The moment you send, run, or present AI work, it's your work — the AI is not an accountable party. If you can't explain it, don't ship it. "The AI wrote it" has never once ended well in a review.

How to Learn

A plan and approach to learn AI — properly

Getting good at AI isn't a course you finish; it's a habit you build. Learn on your real work, in small daily reps, and share what works.

The approach in one line

Learn by doing on your real tasks, a little every day, and pass on what works.

First decision: is this even an AI task?

The most common mistake isn't a bad prompt — it's the wrong tool class for the job. Thirty seconds of task selection beats thirty minutes of prompt iteration.

Sounds like…RouteWhy
"Summarise this recon report for the client call."Generative AIYou're supplying the material; the model transforms it. Home turf.
"What's our leave carry-forward policy?"Search firstThe answer exists in a system. A model without access will invent a plausible policy.
"Total these order adjustments by region."DeterministicModels predict text, they don't compute. Have AI write the SUMIFS or SQL — let Excel or the database do the math.
"Draft the escalation email, firm but polite."Generative AIJudgment plus language transformation — exactly what it's for. Then verify the commitments.

Where the value compounds

1

Personal speed

Drafts, summaries, explanations. Real but bounded — minutes saved per task. Where everyone starts; not where it should end.

2

Workflows — aim here

A recurring multi-step process done with AI end-to-end: notes → MoM → actions → status update. The best return for the effort.

3

Systems

Agents automating whole processes. Highest ceiling, highest care — guardrails scale with autonomy. Earn it after Level 2 is boring.

The trap

Don't mistake Level-1 quick wins for transformation — and don't jump to Level-3 agents before Level-2 workflows are routine. The middle level is where AI quietly pays for itself.

The learning loop

  • Learn — one thing at a time

    One technique or concept. Don't try to learn everything before using anything.

  • Apply — on real work, same day

    Use it immediately on an actual task, not a toy example.

  • Reflect — what worked, what didn't

    Keep the prompts that worked in a notes file.

  • Share — teach a teammate

    Explaining it cements your understanding and lifts the team.

The 5-element prompt framework

ElementWhat it is · example
1 · RoleWho the AI should be. "You are a professional business writing coach…"
2 · TaskExactly what to do. "…review the following email and rewrite it…"
3 · ContextBackground, data, audience. "…it's going to my manager about a project delay…"
4 · FormatThe output shape. "…Subject line, 3-paragraph email, a call to action."
5 · ConstraintsLimits & tone. "…under 150 words; professional but warm."
❌ Weak prompt
Write me an email
✅ Strong prompt

You are a professional business writer. Write a concise follow-up to my manager, Priya, about a delayed task (monthly report, due yesterday, ready by EOD today). Tone: professional, brief apology. Format: subject line + 3 short paragraphs. Under 100 words.

Week 2 of 8 · Foundations

Prompting Techniques & Iteration

Outcome — choose the right technique and refine to a perfect output.

Zero / Few / Chain-of-thoughtSummary · template · improve promptsIteration mindsetCommon mistakes

Three core techniques cover almost everything — and the real skill is iteration: your first output is a draft, not the final answer. The best users refine 2–4 times.

TechniqueHow it works · best for · example
Zero-shotAsk directly, no examples. Simple, well-defined tasks. "Summarise this paragraph in 2 sentences."
Few-shotGive 2–3 examples first. Matching a format/tone/style. "Here are 2 emails I like — write a 3rd in the same style."
Chain-of-thoughtAsk it to reason step by step. Complex decisions, multi-step analysis. "Think step by step: which option should I prioritise and why?"

Common prompt mistakes & the fix

MistakeWhy · the fix
Too vagueNo context/format — add role, context, format, constraints.
Wrong toneTone not set — add "Tone: professional / casual / empathetic."
Output too longNo length limit — add "Under 100 words" or "3 bullets max."
Wrong formatFormat not set — add "Format as a table / numbered list / email."
Made-up factsNo source data — paste your actual data; never ask it to recall specifics.
Ignored a ruleToo many competing instructions — simplify; one clear task per section.
Not actionableTask too broad — break into smaller sub-tasks, one prompt each.

The iteration chain

Run → read → identify what's off (tone? length? format?) → in the same chat say "That was good, but please [specific fix]" → repeat until perfect → save the prompt that worked. That's how your library is born.

Sorting & filtering

  • Sort: single column (Data → A→Z); multi-level (Data → Sort → Add Level: Department, then Name); by colour. Always select the whole table first — never one column alone, or you misalign data.
  • Filter: Ctrl + Shift + L toggles dropdowns. Filter by multiple values (Ctrl-click), or Custom (Number Filters → Greater Than → 50000). Data → Clear before a new filter.
  • Pro tip: press Ctrl + T first — Tables auto-filter, auto-expand, and make formulas read =SUM(EmployeeData[Salary]) instead of =SUM(F2:F21).

Golden ruleWhat it means

Subject-line formula

Reply vs Reply All vs Forward

ScenarioUse
Relevant to everyone on the threadReply All
Only the sender needs itReply
Replying to a mass announcementReply (never Reply All)
Adding a new person / sharing contextForward
Sharing an email chain with someone newForward (don't CC without context)
Correcting a mistake sent to the whole teamReply All (everyone who saw it should see the fix)

Attachments

  • Mention it in the body: "Please find attached the Q3 report."
  • Name files professionally: Q3_Report_Feb2026_v2.pdf — not "report final FINAL.pdf".
  • Large files (>10MB): share a OneDrive/SharePoint link instead.
  • Check it's actually attached before sending — the #1 email mistake.
  • Never send sensitive data (passwords, personal info) as plain text.

Calendar invites & scheduling

  • Title specific ("Weekly Team Sync — Operations", not "Meeting"); set Required vs Optional deliberately; state the timezone; add the Teams link / room.
  • Body = agenda (3–5 bullets) + pre-read + expected duration; reminder 15 min before (1 day for important ones).
  • Recurring: always set an end date (never "no end date"); edit a single occurrence for exceptions; cancel well in advance with a reason.

Prompting

A prompt is just instructions. The gap between a wasted reply and a great one is rarely the AI — it's how clearly you asked. Master the five-part anatomy and a handful of techniques, and your output quality jumps overnight.

Say the job — role, task, context, format

Every prompt · 30 sec

Context Engineering

Prompting is how you ask. Context engineering is making sure the model has the right textbook, calculator, and notes before it starts thinking. The AI knows general SQL; it doesn't know your schema, data, or rules — unless you give them.

Restart when the thread turns against you

Four signals

A long thread accumulates everything — dead ends, rejected drafts, abandoned constraints — all still competing for attention against what you want now. Four signals it's time:

The topic changed

You finished the SQL review and now want an email drafted. The SQL context adds nothing — and can leak tone into the email. New chat.

It keeps returning to a rejected approach

You said no twice and it resurfaces a third time. The rejected idea is baked into the thread. A fresh chat forgets it — you shouldn't have to keep fighting it.

Six-plus iterations without converging

Each correction adds context without removing the wrong turns. Past about six rounds, the thread carries more failed attempts than requirements. Restate cleanly instead of correcting again.

Answers are getting vaguer

Long threads dilute attention — the model hedges more and commits less. Creeping vagueness is the context window telling you it's full of noise.

The exchange rate

One clean restatement in a new chat routinely beats ten more corrections in an old one. Restarting isn't losing progress — the survivors are the progress.

Rolling this out to a team — the enablement layer

Everything so far makes you AI-enabled. This tab is for the moment you're asked to make a team AI-enabled. The consistent finding across organisations: AI adoption fails as a technology rollout and succeeds as a change-management program. Licences don't create capability — structure does.

The enablement model, condensed

Six pillars. Miss one and you'll recognise the failure pattern next to it — most stalled rollouts trace back to a single missing pillar.

1

A named owner

One person accountable for curriculum, tooling, and measurement. Without a DRI, enablement becomes everyone's side project and no one's job. Missing → the initiative fades in six weeks.

2

Visible sponsorship

Leadership repeatedly explains why — in each role's daily terms, not strategy slides. "AI drafts your recon checks so your time goes to the judgment calls." Missing → quiet permission anxiety.

3

Guardrails people can act on

A one-page tiered policy beats a 20-page document. People experiment when they know what's safe; they freeze when they have to guess. Missing → paralysis or shadow AI.

4

Champions, not mandates

Ask for volunteers who are already enthusiastic. Give them this playbook, early access, and a forum. Peer influence beats top-down instruction on adoption, every time. Missing → adoption stalls at the enthusiasts.

5

A shared library

Prompts, Skills, and worked examples in one team space. Hoarded prompts help one person; shared prompts compound across a team. This playbook is the seed. Missing → everyone solves the same problem twice.

6

Measurement that matures

Adoption → engagement → impact, in that order. Don't demand ROI in month one; don't still be counting logins in month six. Missing → the program can't defend its budget.

Build · Foundations

Master prompts in Markdown — write once, build anything

Power users don't type prompts into a chat box and lose them. They write prompts as Markdown (.md) files: structured, reusable, version-controlled, and token-efficient. It is the single most leveraged habit for building apps and agents.

Why Markdown, in one minute

⚡

Fewer, cleaner tokens

Markdown carries structure with almost no syntax overhead. The model reads headings, lists, and code fences natively — so your intent lands with less text than verbose prose.

🔁

Reusable & versioned

A .md file lives in Git: diff it, review it, roll it back, share it. Your prompt becomes a durable asset, not a throwaway message.

🎯

More reliable output

Clear sections — Goal, Features, Rules — remove ambiguity, so the model or the coding agent follows the spec instead of guessing.

$

Lower cost, bigger headroom

Every token is read — and on an API, paid for — and fills a finite context window. Lean Markdown means cheaper calls and more room for real work.

Tokens — the unit you're optimising

AI reads text as tokens, not words — roughly 4 characters ≈ 1 token (about ¾ of a word). Both your prompt and the reply count toward the context window, so leaner input leaves more room for output and lowers cost. Rich formats like .docx or .pdf carry hidden markup that wastes tokens; plain, structured Markdown spends the budget on content.

See your tokens live — the tokenizer

Paste any text and watch it split into tokens with a running count at platform.openai.com/tokenizer. Use it to trim a prompt, compare prose vs Markdown, and estimate cost and context use before you send.

Same intent, a fraction of the tokens

Verbose prose — token-heavy
Hi! I was hoping you might be able to help me out by writing a little script that would go ahead and read in a CSV file for me, and then, if it's not too much trouble, output all of that data as JSON, and please do make sure it handles any errors nicely and doesn't crash...
Structured .md — lean
  • Task: CSV → JSON converter
  • Input: path to a .csv
  • Output: JSON to stdout
  • Rules: handle missing/blank fields; never crash

Same instruction, far fewer tokens — and clearer for the model to follow.

Token-saving habits

  • Structure over prose — headings, lists, and tables instead of long paragraphs.
  • Cut the filler — drop greetings, apologies, and "if it's not too much trouble".
  • State each rule once — repetition costs tokens and confuses priority.
  • Trim examples — two sharp examples beat ten mediocre ones.
  • Reference, don't repeat — point to a section rather than pasting it twice.
  • Drop irrelevant context — give the minimum that makes the answer correct.

A worked example — SKILL.md

# SQL Review

When asked to review SQL, check in this order:
1. Set-based — flag row-by-row cursors and suggest a rewrite.
2. NULL-safe — NULLIF on division, COALESCE on outer joins.
3. Schema-valid — every table/column exists in the given schema.
4. Safety — flag any UPDATE / DELETE / MERGE without a WHERE.

## Output
A short list: issue · why it matters · the fix.

## Boundaries
Do NOT rewrite the whole query unless asked — only report issues.
Never invent column names; if the schema is missing, ask for it.

SQL & Stored Procedures

Where most data teams spend their day — and where AI saves the most time, writing the first draft and the boilerplate so you focus on correctness and performance.

How text-to-SQL actually works

AI doesn't query your database — it writes SQL from your words. The quality hinges on one step: giving it the schema.

Schema awareness is the #1 factor

Give the AI your column lists and it's reliable on standard SELECT / JOIN / GROUP BY. Without them, it invents table and column names that don't exist.

Ask
For sales(region, sale_date, product_id, amount): total revenue per region for Q2 2026, plus each region's share of total, highest first. Set-based, SQL Server 2019.
You get

Aggregation + a window function (SUM() OVER()) — ready to validate. Then refine: "make the quarter a parameter."

Window functions — the concept made visual

The three ranking functions look alike but treat ties differently. Same scores, three different results:

ScoreROW_NUMBER()RANK()DENSE_RANK()
500111
500 (tie)211
48033 (skips 2)2 (no gap)
460443

ROW_NUMBER = always unique · RANK = ties share, then a gap · DENSE_RANK = ties share, no gap. Like Olympic medals: two golds, no silver.

Four worked patterns you'll reuse

-- 1 · Running total (cumulative revenue by month)
SUM(amount) OVER (ORDER BY month
                  ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)

-- 2 · Month-over-month change with LAG()
amount - LAG(amount) OVER (ORDER BY month)   -- prior row

-- 3 · Top N per group (top product in each region)
SELECT * FROM (
  SELECT region, product, sales,
         RANK() OVER (PARTITION BY region ORDER BY sales DESC) rnk
  FROM totals
) t WHERE rnk = 1;

-- 4 · Keep detail AND the group total side by side
amount, SUM(amount) OVER (PARTITION BY region) AS region_total

More asks that work

Customers who purchased in the last 30 days but NOT the prior 6 months.
Total sales by product category for Q1, sorted by revenue desc.
The 2nd-highest salary per department (window function, not TOP).
A pivot: months across columns, product categories down rows.

Apply · For QA

Reconciliation Testing

Compare source to target, prove the numbers tie out, prove nothing was lost or duplicated. After every ETL load you ask one question — can I trust this data? — and answer it with a handful of checks.

The reconciliation flow

The five checks, as SQL

-- 1 · Row count (completeness)
SELECT (SELECT COUNT(*) FROM source) AS src,
       (SELECT COUNT(*) FROM target) AS tgt;

-- 2 · Amount totals (consistency)
SELECT SUM(amount) FROM source;      -- must equal target's

-- 3 · Missing / extra keys (EXCEPT both ways)
SELECT id FROM source EXCEPT SELECT id FROM target;  -- missing
SELECT id FROM target EXCEPT SELECT id FROM source;  -- extra

-- 4 · Duplicate keys within the load (uniqueness)
SELECT id, COUNT(*) FROM target GROUP BY id HAVING COUNT(*) > 1;

-- 5 · Field-level mismatch (FULL OUTER JOIN)
SELECT s.id, s.amount AS src_amt, t.amount AS tgt_amt
FROM source s FULL OUTER JOIN target t ON s.id = t.id
WHERE s.amount <> t.amount OR s.id IS NULL OR t.id IS NULL;

A threshold (e.g. target > 5% below source) can auto-fail the pipeline before bad data spreads.
